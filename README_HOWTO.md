# Избегаем локальный кэш

Это фича Composer: у него есть **встроенный локальный кэш** (`~/.cache/composer/files/` и `~/.cache/composer/repo/`).

Когда вы запускаете `composer install`, Composer сначала проверяет этот локальный кэш. Если архив там есть, он **вообще не делает HTTP-запрос** к вашему прокси. Поэтому прокси "не знает" об этом скачивании и не может показать его в статистике.

Чтобы заставить Composer **всегда** обращаться к вашему прокси (который уже сам выступает в роли кэша), нужно отключить или перенаправить локальный кэш Composer.

Вот самые надежные способы сделать это навсегда, чтобы не писать `--no-cache`.

---

### Способ 1: Глобальная настройка Composer (Рекомендуемый)

Мы не можем просто удалить кэш, но мы можем сказать Composer использовать для кэша пустую временную директорию. Это заставит его каждый раз обращаться по сети (к вашему прокси).

Выполните эти команды в терминале от имени вашего пользователя:

```bash
# 1. Создаем пустую директорию для "фейкового" кэша
mkdir -p /tmp/composer-no-cache

# 2. Глобально указываем Composer использовать её для всех типов кэша
composer config --global cache-files-dir /tmp/composer-no-cache
composer config --global cache-repo-dir /tmp/composer-no-cache
composer config --global cache-vcs-dir /tmp/composer-no-cache
```

**Как это проверить:**
Выполните `composer config --global --list`. Вы должны увидеть:
```text
[cache-files-dir] /tmp/composer-no-cache
[cache-repo-dir] /tmp/composer-no-cache
[cache-vcs-dir] /tmp/composer-no-cache
```

Теперь при любом `composer install` или `composer require` Composer будет вынужден делать HTTP-запросы к вашему прокси `127.0.0.1:8080`. Прокси проверит свой кэш, отдаст `HIT` (мгновенно) или скачает `MISS` (и положит к себе).

NB: НЕ ПРОВЕРЕНО

---

### Способ 2: Через переменную окружения (Идеально для CI/CD или Docker)

Если вы хотите отключить кэш только для определенных сессий (например, на сервере сборки), используйте переменную окружения `COMPOSER_CACHE_DIR`.

Добавьте это в ваш `~/.bashrc` или `~/.zshrc`:
```bash
export COMPOSER_CACHE_DIR=/tmp/composer-no-cache
```
Или передавайте это при запуске команды:
```bash
COMPOSER_CACHE_DIR=/tmp/composer-no-cache composer install
```

---

### Почему не стоит использовать `/dev/null`?

В некоторых старых гайдах советуют делать так: `composer config --global cache-files-dir /dev/null`.
**Не делайте этого.** Composer иногда пытается проверить размер файла или права доступа в этой директории, 
и `/dev/null` может вызвать предупреждения или ошибки типа `Is a directory` или `Permission denied`. 
Пустая реальная папка в `/tmp` работает безупречно и безопасно.

---

### Итоговый рабочий процесс теперь будет таким:

1. Вы запускаете `composer install` (без всяких флагов).
2. Composer видит, что его локальный кэш пуст (или указывает на `/tmp`).
3. Composer делает запрос: `GET http://127.0.0.1:8080/p2/psr/log.json`
4. Ваш прокси отвечает `HIT` (за 2 миллисекунды, читая с диска).
5. Composer делает запрос: `GET http://127.0.0.1:8080/proxy?url=https://api.github.com/...`
6. Ваш прокси отвечает `HIT` (отдавая готовый `.zip` из папки `cache/vendor/package/`).
7. Вы заходите на `/stats` и видите **абсолютно точную картину** того, что и когда было закэшировано.

Это превращает ваш AMPHP-скрипт в **единый источник истины** для кэширования Composer в вашей системе.
